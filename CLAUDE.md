## ⚠️ 최우선 규칙 2: TodoWrite 사용 (무조건 준수)

### TodoWrite 필수 사용 조건
다음 조건 중 **하나라도 해당**되면 작업 시작 전 **반드시** TodoWrite로 할 일 목록 생성:

1. **3단계 이상**의 작업이 필요한 경우
2. **여러 파일** 수정이 필요한 경우
3. **복잡한 작업** (새 기능, 버그 수정, 리팩토링 등)
4. 사용자가 **여러 항목**을 요청한 경우 (콤마, 번호 목록 등)
5. **에이전트 호출**이 필요한 작업

### TodoWrite + 상태 파일 사용 순서 (엄격히 준수)
```
1. 요청 받음 → TodoWrite로 할 일 목록 생성
2. 복잡한 작업이면 → .claude/current-task.md 상태 파일 생성
3. 첫 번째 작업 in_progress로 표시 → 작업 시작
4. 작업 완료 → TodoWrite completed 표시 + 상태 파일 체크리스트 [x] 표시
5. 다음 작업 in_progress로 표시 → 반복
6. 모든 작업 완료 → 작업 완료 보고서 + 상태 파일 삭제
```

### TodoWrite 예외 (사용 안 해도 됨)
- 단일 파일의 1-2줄 수정 (오타, 변수명 변경 등)
- 단순 정보 조회/질문 응답

### 위반 시
- TodoWrite 없이 복잡한 작업 시작 = **규칙 위반**
- 작업 완료 후 completed 표시 누락 = **규칙 위반**

## 작업 규모에 따른 처리 방식

### 직접 처리 (에이전트 사용 안함)
다음과 같은 **간단한 작업**은 에이전트 없이 직접 처리:
- 단일 파일의 간단한 수정 (오타, 변수명, 설정값 변경 등)
- 1~2개 파일에 대한 작은 코드 수정
- 간단한 버그 수정 (원인이 명확한 경우)
- 코드 삭제
- 설정 파일 수정

### 에이전트 사용 (복잡한 작업)
다음과 같은 **복잡한 작업**은 에이전트를 통해 수행:
- 새로운 기능/페이지 구현
- 여러 파일에 걸친 수정
- 아키텍처 설계가 필요한 작업
- 원인을 파악해야 하는 복잡한 버그

### 에이전트 매핑 테이블 (필수 참조)

| 요청 유형 | 사용 에이전트 | 우선순위 |
|----------|--------------|---------|
| 아이디어/기획/설계 | `planner` | 1 |
| 프론트엔드 코드 작성/수정 | `frontend-dev` | 2 |
| 백엔드 코드 작성/수정 | `backend-dev` | 2 |
| UI/UX 디자인/스타일링 | `ui-expert` | 2 |
| 버그 수정/디버깅 | `debugger` | 2 |
| 새 에이전트 생성 | `agent-creator` | - |

### 에이전트 호출 조건

**복잡한 작업**에 해당하면 에이전트 사용:

1. **새 기능/페이지 구현** → planner → 개발 에이전트
2. **복잡한 버그 (원인 파악 필요)** → debugger
3. **기획/설계 요청** → planner
4. **대규모 UI 작업** → ui-expert
5. **여러 파일 수정 필요** → frontend-dev 또는 backend-dev

### 작업 처리 예시

```
# 직접 처리 (에이전트 사용 안함)
사용자: "버튼 색상 파란색으로 바꿔줘" → 직접 수정
사용자: "이 변수명 userId로 바꿔줘" → 직접 수정
사용자: "console.log 삭제해줘" → 직접 수정
사용자: "오타 수정해줘" → 직접 수정

# 에이전트 사용
사용자: "로그인 페이지 만들어줘"
→ planner (기획) → frontend-dev + backend-dev (병렬 개발)

사용자: "왜 버튼이 안 눌리는지 모르겠어요"
→ debugger (원인 파악 필요)

사용자: "결제 시스템 구현해줘"
→ planner → backend-dev + frontend-dev
```

1. 파일 내용 읽기

## 📋 상태 파일 관리 (current-task.md)

### 상태 파일 위치
`.claude/current-task.md`

### 상태 파일 생성 조건
다음 조건 중 하나라도 해당되면 **반드시** 상태 파일 생성:
- 3단계 이상의 작업
- 여러 파일 수정
- 에이전트 호출이 필요한 작업
- 10분 이상 소요 예상 작업

### 상태 파일 형식

```markdown
# 현재 작업: {작업 제목}

## 작업 정보
- **시작 시간**: {YYYY-MM-DD HH:MM}
- **요청 내용**: {사용자 원본 요청}
- **작업 유형**: {기능 구현 / 버그 수정 / 리팩토링 / 기타}

## 체크리스트
- [x] 완료된 작업 1
- [x] 완료된 작업 2
- [ ] 진행 중인 작업 3 ← 현재 진행 중
- [ ] 대기 중인 작업 4
- [ ] 대기 중인 작업 5

## 변경된 파일
| 파일 | 변경 내용 | 상태 |
|-----|----------|------|
| `src/pages/Login.jsx` | 로그인 폼 추가 | ✅ 완료 |
| `src/api/auth.js` | API 함수 | 🔄 진행 중 |
| `src/styles/login.css` | 스타일링 | ⏳ 대기 |

## 컨텍스트 (중요 정보)
- {작업에 필요한 중요 배경 정보}
- {의존성, 제약사항 등}

## 마지막 업데이트
{YYYY-MM-DD HH:MM} - {마지막 완료된 작업}
```

### 상태 파일 업데이트 규칙

| 시점 | 행동 |
|-----|------|
| 복잡한 작업 시작 | 상태 파일 생성 |
| 각 단계 완료 | 체크리스트 [x] 표시 + 마지막 업데이트 갱신 |
| 파일 수정 | 변경된 파일 테이블 업데이트 |
| 모든 작업 완료 | **상태 파일 삭제** |

### 상태 파일 삭제 조건
- 모든 체크리스트 항목 완료
- 작업 완료 보고서 제출 후
- 사용자가 명시적으로 작업 취소

## 핵심 규칙

1. **TodoWrite 우선**: 복잡한 작업 시작 전 반드시 할 일 목록 생성 (최우선 규칙 참조)
2. **상태 파일 관리**: 복잡한 작업 시 `.claude/current-task.md` 생성, 완료 시 삭제
3. **작업 규모 판단**: 간단한 작업은 직접 처리, 복잡한 작업은 에이전트 사용
4. **기능 보존**: 구조 수정 시에도 기존 기능 절대 변경 금지
5. **수동 테스트**: 자동화 스크립트 대신 하나씩 직접 테스트
6. **의도 확인**: 작업 전 사용자 의도 먼저 파악
7. **요청 범위 준수**: 사용자가 요청한 작업만 수행. 요청 외 추가 수정, 리팩토링, 개선 금지
8. **React 빌드 필수**: React 프로젝트(front-end) 작업 완료 후 반드시 `npm run build` 실행
9. **데이터 삭제 승인 필수**: 데이터를 삭제할 때는 반드시 사용자의 승인을 받은 후 삭제

## 작업 처리 프로세스

### 1. 요청 분석 및 처리 방식 선택

```
1. 요청 분석 → 작업 규모 판단
2-a. 간단한 작업 → 직접 처리
2-b. 복잡한 작업 → 에이전트 매핑 테이블 확인 → Task tool로 에이전트 호출
3. 결과 확인 및 보고
```

### 에이전트 실행 순서

```
1. planner (기획이 필요한 경우 먼저)
   ↓
2. 개발 에이전트 (독립적이면 병렬 실행)
   - frontend-dev (프론트엔드)
   - backend-dev (백엔드)
   - ui-expert (디자인)
   ↓
3. debugger (문제 발생 시)
```

### 2. 태스크 관리

**위의 "최우선 규칙: TodoWrite 사용" 섹션 참조**

핵심 원칙:
- 한 번에 하나의 작업만 `in_progress`
- 작업 완료 즉시 `completed`로 표시 (배치 처리 금지)

### 3. React 프로젝트 빌드

프론트엔드(React) 작업 완료 후 반드시 실행:

```bash
cd front-end && npm run build
```

빌드 실패 시 오류를 수정하고 다시 빌드할 것

## 에이전트 호출 전략 (핵심)

### 중요: 에이전트 간 호출 불가

```
⚠️ 에이전트에서 다른 에이전트를 호출할 수 없음
✅ 메인 Claude가 오케스트레이터 역할 → 모든 에이전트 직접 호출
```

### 메인 Claude의 역할

1. **요청 분석** → 어떤 에이전트가 필요한지 판단
2. **TodoWrite** → 작업 목록 생성 및 관리
3. **에이전트 호출** → Task tool로 필요한 에이전트 호출
4. **결과 조합** → 각 에이전트 결과를 통합하여 보고

### 에이전트 호출 규칙

| 요청 유형 | 호출 에이전트 | 호출 방식 |
|----------|--------------|----------|
| 새 기능 기획 | `planner` → 결과 확인 → `frontend-dev` + `backend-dev` | 순차 |
| 프론트+백엔드 개발 | `frontend-dev` + `backend-dev` | **병렬** |
| UI 디자인 + 개발 | `ui-expert` → `frontend-dev` | 순차 |
| 버그 수정 | `debugger` | 단독 |
| 프론트만 수정 | `frontend-dev` | 단독 |
| 백엔드만 수정 | `backend-dev` | 단독 |

### 병렬 호출 방법 (필수)

**독립적인 작업은 반드시 병렬로 호출**하여 효율성 극대화:

```
# 병렬 호출 예시 - 하나의 메시지에 여러 Task 호출
Task(subagent_type="frontend-dev", prompt="로그인 페이지 구현")
Task(subagent_type="backend-dev", prompt="로그인 API 구현")
```

### 순차 호출이 필요한 경우

```
# 순차 호출 - 앞 작업 결과가 필요한 경우
1. planner 호출 → 기획 결과 받음
2. 기획 결과를 바탕으로 frontend-dev + backend-dev 병렬 호출
```

### 에이전트별 역할 (독립 수행)

| 에이전트 | 역할 | 스킬 참조 |
|---------|------|----------|
| `planner` | 기획, 설계, 개발 방향 정리 | planner |
| `frontend-dev` | React 컴포넌트, CSS 구현 | frontend-dev |
| `backend-dev` | Express API, Prisma, S3 | backend-dev |
| `ui-expert` | UI/UX 디자인, 스타일링 가이드 | ui-expert |
| `debugger` | 버그 원인 파악, 수동 테스트 | debugger |
| `agent-creator` | 새 에이전트 생성 | - |

### 에이전트 호출 시 표시

```markdown
## 🤖 에이전트 실행 중

| 에이전트 | 상태 | 작업 내용 |
|---------|------|----------|
| frontend-dev | 🔄 진행 중 | 로그인 페이지 컴포넌트 생성 |
```
